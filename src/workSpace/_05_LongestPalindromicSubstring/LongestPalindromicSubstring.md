# 5. Longest Palindromic Substring
   Given a string s, return the longest palindromic substring in s.


----
Example 1:
Input: s = "babad"
Output: "bab"
Explanation: "aba" is also a valid answer.

Example 2:
Input: s = "cbbd"
Output: "bb"


---
My code's problem
시진님, 이전 피드백에서 배운 StringBuilder와 subList를 바로 응용해서 코드를 작성하신 모습에 정말 감탄했습니다! 배운 것을 자기 코드로 녹여내는 능력이 아주 탁월하십니다.
하지만 안타깝게도 이 코드는 리트코드에서 에러(Exception)를 뿜거나 틀린 답을 낼 수밖에 없는 4가지 치명적인 논리적/문법적 함정이 숨어있습니다. 왜 어레이 리스트를 쓰면 안 되었는지, 그리고 코드의 어느 부분이 발목을 잡았는지 아주 냉정하고 명확하게 분석해 드릴게요.
🚨 1. 영원히 돌아오지 않는 confirm (변수 스코프의 함정)
가장 기초적이지만 가장 많이 하는 실수입니다.
boolean confirm = true; // 반복문 밖에 선언됨
for (int i=0; i<s.length(); i++) { ... }

confirm을 밖에서 true로 선언해 두셨습니다. 만약 반복문을 돌다가 팰린드롬이 아닌 것을 발견해서 confirm = false;가 딱 한 번이라도 실행되면 어떻게 될까요?
그 뒤로는 confirm이 다시 true로 돌아갈 방법이 없습니다. 즉, 첫 번째 실패 이후에 등장하는 진짜 팰린드롬들은 if (confirm && length > max) 조건을 절대 통과하지 못하고 전부 무시당하게 됩니다.
• 해결법: confirm은 매번 팰린드롬을 검사할 때마다 true로 새롭게 초기화되어야 합니다. (반복문 안쪽으로 위치 이동)
🚨 2. "초기화(clear)"가 낳은 참사 (알고리즘 논리 오류)
가장 치명적인 알고리즘 논리 오류입니다.
list.clear();

시진님은 팰린드롬을 하나 찾고 나면 바구니를 싹 비워버렸습니다. 이것은 **"다음 팰린드롬은 무조건 앞의 팰린드롬이 끝난 다음부터 시작할 것이다"**라는 잘못된 가정을 한 것입니다.
• 반례 ("abacab"): 1. 처음에 "aba"를 찾습니다. max=3을 기록하고 리스트를 비웁니다(clear). 2. 이제 남은 글자인 "cab"를 리스트에 담기 시작합니다. 3. 영원히 가장 긴 정답인 "bacab"(길이 5)는 찾을 수 없습니다. 이미 앞부분의 b와 a를 쓰레기통에 버렸기 때문입니다.
• 결론: 팰린드롬 문제는 글자들이 서로 '겹쳐서(Overlapping)' 나타날 수 있기 때문에, 원본 문자열을 절대 훼손하거나 날려버리면 안 됩니다.
🚨 3. 폭탄을 품은 인덱스 계산 (IndexOutOfBoundsException)
if (list.get(i-p) == list.get(0) && i != 0)

리스트를 clear()로 비우고 나면 리스트의 size()는 0이 됩니다. 하지만 전체 문자열을 도는 i는 계속 증가하죠.
만약 i가 5인데 리스트를 방금 비워서 요소가 1개밖에 없다면? list.get(5 - p)를 하는 순간 **"리스트에 그런 번호는 없는데?"**라며 프로그램이 즉사(에러)하게 됩니다.
🚨 4. 객체 비교의 함정 (== 와 !=)
if (list.get(j) != list.get(length-j-1))

list.get()이 반환하는 것은 기본형 char가 아니라 껍데기가 씌워진 객체인 **Character**입니다.
자바에서 객체끼리 ==나 !=로 비교하면 "내용물이 같아?"가 아니라 **"메모리 주소(껍데기)가 같아?"**라고 묻는 것입니다. (ASCII 코드 범위 내에서는 자바가 캐싱을 해서 우연히 작동할 수도 있지만, 본질적으로 매우 위험한 코드입니다.)
• 해결법: .equals()를 쓰거나 (char) list.get(j) 처럼 기본형으로 벗겨낸(Unboxing) 뒤에 비교해야 안전합니다.
💡 총평: 왜 ArrayList를 쓰면 안 좋았을까?
결론적으로 이 문제에서 ArrayList를 쓰는 것은 **"이미 잘 차려진 밥상(String)을 놔두고, 굳이 새 그릇(ArrayList)에 음식을 하나씩 옮겨 담으면서 먹으려는 행동"**과 같습니다.
자바의 String은 그 자체로 이미 문자들의 배열입니다.
시진님이 list.get(j)로 하고 싶었던 것은 그냥 s.charAt(j)로 다 할 수 있습니다. 굳이 리스트에 글자를 하나씩 add하고, 다 찾으면 clear하고, 다시 문자열로 합치는(StringBuilder) 과정들은 컴퓨터의 메모리와 시간을 엄청나게 낭비하게 만듭니다.
시진님의 접근법(처음부터 끝까지 훑으면서 확인한다)을 유지하면서 정답을 맞히려면, 리스트를 아예 없애버리고 인덱스 번호(int)들만 가지고 원본 문자열 s 위를 왔다 갔다 하며 비교해야 합니다. 그것이 바로 이전 답변에서 보여드렸던 O(N^2) 속도의 '중심 확장(Expand Around Center)' 알고리즘입니다.
